# 审：金字塔 → 四色验证树 + 底稿

你收到"读"技能产出的重组金字塔。对每个数据节点和因果边独立验证，标色，留底稿。

---

## 本技能不做的事

- 不拆解或重组金字塔（那是"读"技能）
- 不合并多篇报告（那是"合"技能）
- 不添加新论点。只验证已有的。

---

## 第 1 步：节点分类

遍历整棵树，对每个节点分类：

| 问自己 | → 分类 | 颜色规则 |
|--------|--------|---------|
| 有子节点，且标题是纯分类标签（"供给端"、"需求端"）？ | **结构节点** | 永远黑色 #333，不验证 |
| 包含"常识"或"Common Knowledge"？ | **常识节点** | 永远黑色 #333，不验证 |
| 有子节点，且标题是 Minto 结论句？ | **结论标题** | 验证逻辑，不验证数据。根据结论是否成立标色 |
| 叶节点，包含数据/事实/声明？ | **数据节点** | 验证数据。根据独立来源是否确认标色 |

产出：所有节点的分类清单 + 哪些需要验证。

---

## 第 2 步：识别因果边

"边"是两个节点之间的因果关系："A 导致 B" 或 "A 支持 B"。

对树中的每对父子关系，问：
→ 这里有隐含的因果声明吗？（如"供需缺口 → 价格上涨"）
→ 有：这条边需要验证
→ 没有：这只是结构分组，跳过

产出：需要验证的边的清单。

---

## 第 3 步：确定验证时点

**核心规则：以报告发布时点为准验证，不是以今天为准。**

1. 从元数据读取报告发布日期
2. 每次搜索验证时，限定在该日期附近

3. 当你搜到的当前数据与报告不同时：

→ 报告的数据在发布时是准确的吗？
  → 是：颜色 = **绿色**。附 UPDATE 说明当前变化。
  → 不是：颜色 = **红色**（数据在当时就是错的）。

**举例：**
- 报告（2025.12）写："FedWatch 显示 2026 年将降息两次"
- 你搜索（2026.5）发现：FedWatch 显示 70% 概率不降息
- 你搜 2025 年 12 月的 FedWatch 历史数据，发现确实显示预期降息两次
- 结果：**绿色** + UPDATE："截至 2026.5，市场预期已转向不降息（70% 概率）"

**常见错误：**
- 因为"现在不对了"就标红——那是 UPDATE，不是错误
- 没有搜报告时点的历史数据
- 用今天的眼光评判一个在当时合理的预测

---

## 第 4 步：验证数据节点（每个节点一个 Sub Agent）

对每个数据节点，执行以下序列：

### 4.1 提取声明

写成一句可证伪的陈述：
- ❌ "白银供给在变化"
- ✅ "2025 年全球矿产银产量 846.6 Moz（来源：报告第 36 页）"

### 4.2 搜索一手来源

用 `web_search` 找报告引用的**原始来源**，不是二手转述。

搜索策略：
1. 搜具体数字 + 来源名（如 "846.6 Moz silver 2025 Silver Institute"）
2. 没结果 → 搜来源机构 + 主题（如 "Silver Institute World Silver Survey 2026"）
3. 还没结果 → 搜数字本身在任何权威来源中的出现

### 4.3 抓取并比对

用 `web_fetch` 打开最可能的 URL。比对：

| 检查 | 结果 |
|------|------|
| 数字完全一致 | 强证据 → 绿色 |
| 数字在舍入范围内一致（846.6 vs 847） | 绿色 + 注明 |
| 数字一致但单位/时间范围不同 | 黄色 — 检查报告是否正确转换 |
| 数字明显不同 | 先查验证时点，再决定红色 |
| 完全找不到来源 | 黄色（有来源但无法独立验证） |

### 4.4 标色

| 颜色 | 色值 | 标准 |
|------|------|------|
| **绿** | #009A44 | 独立来源在报告发布时点确认了该声明 |
| **金** | #D4A843 | 报告引用了来源但你无法独立验证（URL 失效、付费墙等） |
| **红** | #CC0000 | 独立来源与该声明矛盾（即使在报告发布时点） |

### 4.5 记录底稿

```json
{
  "node_id": "T1.2",
  "type": "data_node",
  "claim": "2025年全球矿产银产量846.6Moz",
  "original_text": "2025年全球矿产银产量为846.6百万盎司",
  "original_source": {"report": "guotai", "page": 36},
  "color": "green",
  "sources": [
    {
      "url": "https://实际找到的URL",
      "title": "World Silver Survey 2026",
      "finding": "确认：矿产银产量 846.6 Moz（2025年）"
    }
  ],
  "update_note": null,
  "subclaims": [
    {"claim": "子声明1", "verified": true, "source_url": "..."},
    {"claim": "子声明2", "verified": false, "source_url": "..."}
  ]
}
```

**来源规则：**
- 每个来源**必须有可点击的 URL**。没有 URL → 删除该来源
- 绝不编造 URL。找不到 → 标黄色
- 报告原文引用必须保持原始语言。中文报告的引用不得翻译成英文呈现

---

## 第 5 步：验证因果边

对每条边（因果关系），验证的是**逻辑**，不只是两端的数据。

### 5.1 提取因果声明

写成："A 导致/引起/暗示 B，因为[机制]"
- ❌ "供给和价格有关"
- ✅ "持续供需缺口导致库存消耗，当仓库库存低于可交割合约量时触发逼空风险"

### 5.2 搜索证据

搜索三个方面：
1. 声称的机制——它真的按报告说的那样运作吗？
2. 反例——这个机制有没有失效过？
3. 关系强度——是直接因果、相关性、还是猜测？

### 5.3 标色

| 颜色 | 标准 |
|------|------|
| **绿** | 因果机制有充分证据支持、学界/业界公认 |
| **金** | 机制合理但找不到强独立确认 |
| **红** | 机制被证据反驳，或关键假设不成立 |

---

## 第 6 步：验证结论标题

对每个 Minto 结论句标题（有子节点、标题是结论不是标签）：

1. 读所有子节点
2. 问：结论能从子节点逻辑推出吗？
   → 能：标题继承子节点中**最好**的颜色
   → 不能：标题过度概括——标红或标黄并附注

---

## 第 7 步：颜色传导

所有单独验证完成后，向上传导颜色：

**规则：父节点颜色 = 子节点中最好的（绿 > 金 > 红）**

- 有一个子节点是绿 → 父节点至少是绿
- 黑色节点**不参与**传导
- 常识节点**不参与**传导

**为什么取最好（不是最差）？** 一个主干有一个强绿色数据点和一个弱黄色点，比全黄的主干支撑力更强。绿色节点锚定了论证。

---

## 第 8 步：最终质检

审查完整的染色树：

| 检查 | 不通过则 |
|------|---------|
| 有红色节点应该在时点修正后变绿吗？ | 用报告日期数据重新验证 |
| 有数据节点还没标色吗？ | 至少标黄 |
| 每个底稿都有至少一个带 URL 的来源吗？ | 没有 → 降为黄色 |
| 中文报告的引用还是中文吗？ | 被翻译了 → 改回原文 |

---

```json
{
  "report_id": "string",
  "report_publish_date": "YYYY-MM-DD",
  "verification_date": "YYYY-MM-DD",
  "summary": {"total_nodes": 0, "green": 0, "gold": 0, "red": 0, "black": 0},
  "verifications": [
    {
      "node_id": "string",
      "type": "data_node|causal_edge|conclusion_heading",
      "claim": "string",
      "original_text": "string (报告原文)",
      "original_source": {"report": "string", "page": 0},
      "color": "green|gold|red|black",
      "sources": [{"url": "string", "title": "string", "finding": "string"}],
      "update_note": "string 或 null",
      "subclaims": [{"claim": "string", "verified": true, "source_url": "string"}]
    }
  ]
}
```
